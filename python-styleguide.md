# Python Style Guide

Standards for Python code. These apply to every change, whether written by a human, generated by an AI tool, or a mix.

> **Maintaining this document:** Keep it direct, concise, and actionable. Do not add filler, over-explain concepts, or pad sections for completeness. If a rule can be stated in one sentence, use one sentence. If a section stops being relevant, delete it. This document should be as lean as the code it governs.

---

## Why These Guidelines Exist

AI coding tools (Claude, Cursor, Copilot, etc.) are now part of the development workflow. They increase the speed at which code is produced, but speed without standards produces slop. AI-generated code trends toward verbose, loosely-typed, convention-ignoring output unless it is held to explicit rules.

Without strict guardrails:
- Env vars get accessed directly via `os.getenv` instead of typed config
- Functions grow to thousands of lines because the tool doesn't know when to stop
- Business logic lands in I/O and orchestration layers because the tool doesn't understand the architecture
- Untyped dicts proliferate because the tool defaults to the path of least resistance
- Tests mock everything instead of using clean dependency boundaries
- Every file gets a module-level docstring and every function gets a paragraph that restates the code
- Existing utilities and patterns get reimplemented because the tool doesn't know what's already there

These guidelines are not aspirational. They are the minimum standard for code entering a repo.

---

## 1. Module Size and Structure

**No multi-thousand-line files.** If a module exceeds ~300-500 lines, it should be broken into logical components.

Large functions are a code smell. If a function exceeds ~50 lines, extract sub-functions. If a module is growing because it handles multiple responsibilities, split it into separate modules grouped by concern.

```
# Bad: one 800-line module doing parsing, validation, transformation, and persistence
order_processor.py -> process_orders() handles everything

# Good: each concern in its own module
parsing.py         -> parse_order(), extract_line_items()
validation.py      -> validate_schema(), check_required_fields()
transform.py       -> normalize_fields(), apply_discounts()
persistence.py     -> save_order(), archive_order()
```

## 2. Comments and Documentation Noise

**Every function should have a docstring.** Scale detail to complexity -- one-liner for simple helpers, longer for non-obvious behavior. Don't restate the function name and signature in paragraph form.

```python
# BAD -- excessive docstring that restates the signature
def get_order(self, order_id: str) -> Order:
    """Get an order by its ID.

    Args:
        order_id: The ID of the order to get.

    Returns:
        An object containing the order data.
    """
    return self._get(f"/orders/{order_id}")

# GOOD -- simple one-liner for a simple function
def get_order(self, order_id: str) -> Order:
    """Fetch a single order by ID."""
    return self._get(f"/orders/{order_id}")

# GOOD -- detailed docstring for complex behavior
def archive_order(self, order_id: str, reason: str) -> ArchivedOrder:
    """Move an order to the archive store, then soft-delete the live record.

    The archive key is deterministic based on order_id and the original
    created-at date. Returns the archived record; raises OrderNotFound if
    the live record is already gone.
    """
    ...
```

**Rules:**
- Do not add module-level docstrings unless the module's purpose is genuinely non-obvious from its name and contents
- Do not add inline comments that restate the next line of code
- Do add comments when the *why* is not obvious -- business rules, non-obvious constraints, workarounds
- Docstring detail should scale with function complexity, not be a fixed template

## 3. Environment Variables and Config Propagation

**All environment variables must flow through a typed Pydantic `BaseSettings` model.** Never use `os.environ`, `os.getenv`, or any direct env var access.

```python
# WRONG -- untyped, unvalidated, scattered
enabled = os.getenv("IS_ENABLED", "false").lower() == "true"
url = os.environ["API_URL"]

# RIGHT -- typed, validated, single source of truth
class AppConfig(BaseSettings):
    model_config = SettingsConfigDict(extra="forbid")

    notifications_enabled: bool = Field(
        default=False,
        validation_alias="IS_ENABLED",
    )
    api_url: str = Field(
        validation_alias="API_URL",
    )
```

**Config propagation:** `get_settings()` is called **once at the entry point** (app lifespan, CLI main, worker startup, script). Everything below the entry point receives the config object as an explicit, required constructor parameter.

```python
# WRONG -- lib reaches for global singleton
class Processor:
    def __init__(self):
        self._config = get_settings()  # hidden dependency

# WRONG -- optional config with fallback
class Writer:
    def __init__(self, config=None):
        self._config = config or get_settings()  # unclear who owns config

# RIGHT -- config is required, passed by caller
class Writer:
    def __init__(self, config: AppConfig) -> None:
        self._config = config
```

Do not create derived config models that duplicate fields from the settings class. One config class, one source of defaults. If a component only needs a subset of fields, accept the full config and use what it needs -- do not copy fields into a second model.

**Why:**
- Env vars are validated at startup, not at runtime
- Types are validated automatically (no manual string parsing)
- All config is visible in one place
- Testable via `config.model_copy(update={...})`
- Explicit dependency makes it clear where config comes from and easy to override in tests

## 4. Data Models

**All structured data must use Pydantic `BaseModel`.** No `TypedDict`, no raw `dict[str, Any]` for structured data, no mutable dataclasses.

```python
class OrderRequest(BaseModel):
    model_config = ConfigDict(strict=True, extra="forbid")
    source: Literal["web", "api"]
    line_items: list[LineItem]
    customer_id: str
```

Baseline defaults:
- `strict=True` -- no silent type coercion, unless explicitly desired
- `extra="forbid"` -- reject unknown fields (catches typos)

Prefer immutability with `frozen=True` for data passed between stages and for value objects -- it prevents accidental mutation. Opt out only when a model is genuinely meant to be mutated (accumulators, builders, records populated incrementally).

**When to create a model:** If data is passed between functions, returned from a function, or received from an external source, it gets a model. If you're passing a `dict[str, Any]` and accessing keys by string, that's a model waiting to be defined.

## 5. Separation of Concerns

**Keep business logic separate from infrastructure and orchestration.** Business logic should be testable without spinning up infra, and infra concerns (retry, caching, state, I/O) should stay out of business logic.

Two rules carry most of the weight:

1. **Pure logic knows nothing about its callers or its I/O.** No framework imports, no network/disk/database calls inside functions that compute or decide. Those functions take data in and return data out.
2. **Imports point one direction -- toward pure logic.** Entry points and adapters depend on business logic; business logic never depends on them.

```
entrypoints/   -> app/CLI/worker wiring, calls into services   (may import everything below)
adapters/      -> I/O: HTTP clients, DB, storage, queues       (imports core + models)
core/          -> pure business logic                          (imports models only)
models.py      -> data contracts (Pydantic)
config.py      -> BaseSettings + app configuration
```

This is one illustration, not a mandated tree -- libraries, CLIs, and services each shape it differently. What must hold everywhere: the dependency arrows point one way, and pure logic sits at the bottom with no upward or outward imports. If you find yourself adding a lazy import to dodge a circular dependency, the layering is wrong -- fix the structure instead.

A shared/utility layer (e.g. `shared/`) may be imported by anything, but must never import from a specific feature or service.

## 6. Error Handling

**Every distinct failure mode gets a named exception class.** Distinguish between transient (retryable) and permanent (domain) errors.

```python
# Transient -- worth retrying; same call may succeed later
class UpstreamServiceError(Exception): ...  # 5xx, timeout, connection error
class StorageError(Exception): ...          # transient backend failure

# Permanent -- same input will produce the same failure
class ValidationError(Exception): ...       # input cannot be parsed
class OrderNotFound(Exception): ...         # nothing to act on
```

**Rules:**
- Avoid catching and swallowing exceptions silently (`except Exception: return []`). In rare cases this is acceptable (e.g., best-effort enrichment that should not block the main flow), but it must be logged and the rationale documented.
- Never hand-roll retry loops with `time.sleep()`. Use a dedicated retry library (e.g. `tenacity`) or your framework's retry facility, and apply it only to the transient exceptions.
- Let transient exceptions propagate to the retry layer; don't retry deep inside business logic.
- Raise domain exceptions for permanent failures so they surface immediately instead of being retried pointlessly.

## 7. Testing

**Tests must not require external infrastructure.** No port-forwarding, no live databases, no real cloud storage.

- **Config via fixture:** Use a shared `conftest.py` fixture. Override settings with `config.model_copy(update={...})` -- never `os.environ` in tests.
- **Prefer stubs over mocks:** Where possible, inject lightweight stub classes rather than patching internals with `unittest.mock.patch`. Reserve mocks for external library calls that can't be injected.
- **Test logic directly:** Most tests should target business logic functions (pure Python). Reserve heavier tests for adapters (I/O behavior) and end-to-end wiring.

```python
# conftest.py
@pytest.fixture()
def config() -> AppConfig:
    return AppConfig(
        api_url="http://api/",
        storage_root="test-store",
    )

# test file
def test_notifications_skipped_when_disabled(config):
    cfg = config.model_copy(update={"notifications_enabled": False})
    assert not should_notify(order, cfg)
```

## 8. What to Actively Avoid

| Anti-pattern                                   | Do this instead                                  |
| ---------------------------------------------- | ------------------------------------------------ |
| `os.environ` / `os.getenv`                     | Pydantic `BaseSettings` field                    |
| `TypedDict` for structured data                | Pydantic `BaseModel`                             |
| `dict[str, Any]` as a function return type     | Named Pydantic model                             |
| Business logic in orchestration/I/O layers     | Logic in pure functions; adapters delegate to it |
| Manual retry loops with `time.sleep`           | A retry library (`tenacity`) or framework retries |
| Silent `except Exception: pass`                | Named exceptions + logging (or document why)     |
| 500+ line modules                              | Split by responsibility                          |
| 100+ line functions                            | Small functions, single responsibility          |
| Moving data through your process to relocate it | Let the storage layer copy/move server-side      |
| `get_settings()` inside a lib or mid-execution | Require config as a constructor parameter        |
| `config or get_settings()` fallback            | Make config a required param, no default         |
| Derived config models duplicating fields       | Accept the main settings class directly          |
| Hardcoded env var strings in tests             | `config.model_copy(update={...})`                |
| Lazy imports to work around circular deps      | Fix the module structure                         |
| Verbose docstrings that restate the signature  | Brief one-liner, or detailed for complex logic   |
| Module-level docstrings on every file          | Only when purpose isn't clear from name/contents |
